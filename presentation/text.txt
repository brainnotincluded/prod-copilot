=== СЛАЙД 3: ПРОБЛЕМА ===
У большинства внутренних платформ всё нужное уже есть через API
Но собрать из этих возможностей рабочий сценарий — дорого:

Прыжки между системами
Чтобы решить одну задачу — нужно зайти в 3-5 разных систем и вручную склеить данные

Всё держится в голове
Нужно помнить, какие API что умеют, какие сущности как связаны и в каком порядке вызывать

Чёрный ящик исполнения
Непонятно, что уже выполнено, где ошибка и можно ли доверять результату без разработчика рядом

Итог: высокий порог входа, медленное выполнение и постоянный риск ошибки при ручной оркестрации

=== СЛАЙД 5: АРХИТЕКТУРА ===
5 сервисов в Docker Compose:

ML LAYER — Оркестратор
- Разбор бизнес-запроса
- Построение плана шагов
- Выбор действий из capabilities
- Reasoning и альтернативы
- Контекст = описания API
- Обработка неопределённости

BACKEND LAYER — Исполнитель
- Оркестрация вызовов API
- Контроль ролей и прав
- Approval для state-changing
- Логирование каждого шага
- Модель данных и состояния
- Декларативное подключение API

UI LAYER — Интерфейс
- Чат с оркестратором
- Карта API и возможностей
- Визуальный граф сценария
- Авто-интерфейс результата
- История запусков
- Экран согласования

Схема:
Frontend (Vue 3) <-- WebSocket --> Backend (FastAPI) <-- HTTP/SSE --> MLOps (FastAPI)
PostgreSQL + pgvector — хранение эндпоинтов и эмбеддингов
External APIs — JSONPlaceholder, Marketing API, любые другие

=== СЛАЙД 6: КАК ЭТО РАБОТАЕТ ===
1. Загрузка Swagger спецификации (файл или URL)
2. Парсинг эндпоинтов + генерация эмбеддингов
3. Пользователь пишет запрос в чате
4. LLM классифицирует intent (чат vs API запрос)
5. Planner создаёт план выполнения
6. Executor выполняет шаги: api_call -> data_process -> execute_code -> format_output
7. Результат: таблица / график / карта / изображение

5 фаз продукта (MECE):
01 Понимание возможностей — подключённые API, capabilities, роли, карта сущностей
02 Постановка задачи — бизнес-запрос в чате, уточнение, перевод в структуру
03 Планирование сценария — пошаговый план, выбор API, проверка исполнимости
04 Безопасное исполнение — approval state-changing, оркестрация, статусы и ошибки
05 Проверка результата — рабочий интерфейс, KPI и графики, история и повтор

=== СЛАЙД 7: КАРТА API ===
Vue Flow визуализация:
- Группировка по API источникам (фоновые контейнеры)
- Кластеры ресурсов (users, posts, segments)
- Collection <-> Detail связи (/{id})
- FK связи (userId -> users) — пунктирные стрелки
- Parent ресурсы (/posts/{id}/comments -> /posts)
- Легенда: цвета методов + типы связей

Что видит пользователь:
- Все подключённые API и их возможности
- Какие действия доступны именно ему (по роли)
- Read / Write разграничение
- Визуальная схема связей между сущностями
- Подсказки по комбинированию блоков

Зачем это важно:
Пользователь видит реальные границы возможного до старта сценария —
система не обещает невозможного и не создаёт ложных ожиданий.
Новые capabilities подключаются декларативно — загрузить Swagger и готово.

=== СЛАЙД 14: БЕЗОПАСНОСТЬ И КОНТРОЛЬ ===
- URL allowlisting: executor блокирует вызовы вне base_url
- SSL: verify=False для Docker (configurable)
- Подтверждение действий: POST/PUT/DELETE требуют confirmation
- Pydantic field_validator: result всегда dict|None
- Context truncation: защита от переполнения LLM контекста
- Code sandbox: subprocess с timeout, изолированная директория

E2E демо сценарий:
Маркетолог описывает задачу в чате -> система собирает E2E маршрут из 6 шагов:
1. Поиск сегментов: Премиум + Неактивные 60+ + Высокий чек
2. Прогноз аудитории для разных логик пересечения
3. [!] Создание аудитории — approval
4. Генерация 2-3 вариантов push-уведомления
5. [!] Создание черновика кампании — approval
6. Авто-интерфейс: KPI, график, карточки, журнал

Домен — пример, не ограничение. Система работает для любой платформы с описанными API.
